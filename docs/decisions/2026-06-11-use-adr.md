# ADR: Use Architecture Decision Records

**Status:** Accepted
**Date:** 2026-06-11
**Tags:** meta

## Context
Aisy is a public, open-source agent harness — an "OS" wrapping a stateless LLM
"CPU". Its architecture carries consequential, non-obvious tradeoffs: a
file-based memory paradigm over vectors, deterministic code hooks over prompt
rules, bi-temporal soft-delete to fix the memory-resurrection bug, MCP tool-
poisoning defenses, provider routing with hysteresis fallback, and KV-cache
discipline. Contributors who arrive later — or the maintainer six months on —
need to understand *why* each choice was made before they can safely extend or
challenge it. Rationale that lives only in commit messages, chat, or someone's
head is lost the moment context shifts. A public OSS project, where strangers
submit PRs and audit security claims, especially needs a durable, reviewable
decision trail kept inside the repo and versioned alongside the code it governs.

## Decision
This project records every consequential architectural decision as a MADR-3.0
Architecture Decision Record in `docs/decisions/`. Files are named
`YYYY-MM-DD-kebab-slug.md`, each carries a logical id `ADR-NNNN` in its title,
and an `INDEX.md` lists all ADRs latest-first.

What qualifies as consequential: technology/framework/runtime/language choices,
license decisions, packaging/distribution model, long-lived architectural
patterns, major dependency adoption or drop, deprecation/supersedure/breaking
changes, and security/privacy tradeoffs. What does not: bug fixes, tactical
refactors with no architectural impact, config tweaks, and temporary stubs.

Each ADR follows the fixed section layout — Context, Decision, Consequences
(Positive/Neutral/Negative), Alternatives considered, References — and cross-
links related ADRs by relative path. An ADR may be superseded but is never
edited away; the trail is append-and-supersede, mirroring how the harness
itself treats memory.

## Consequences
- **Positive:** Every consequential change ships with the reasoning behind it,
  reviewable in the same PR; newcomers can reconstruct the architecture's
  intent; security and design claims become auditable by outsiders; superseded
  decisions leave a visible history instead of vanishing.
- **Neutral:** Maintaining `INDEX.md` latest-first and assigning `ADR-NNNN` ids
  becomes part of the normal change ritual; ADRs live in git and travel with
  the code under Apache-2.0.
- **Negative:** Small overhead per consequential change; risk of "ADR theater"
  if records are written after the fact to rubber-stamp a done deal rather than
  to think; judgment is required at the consequential/tactical boundary.

## Alternatives considered
**Undocumented decisions** — fastest in the moment, but rationale evaporates;
this is precisely the failure a reviewable OSS harness cannot afford, since
contributors would re-litigate settled tradeoffs blind. Rejected.

**Project wiki / external knowledge base** — searchable, but drifts out of sync
with the code, is not versioned with the change that motivated it, and is
invisible in PR review. Rejected.

**Design-docs only** — good for exploring a space, but they describe *what* was
built without a discrete, dated, supersedable record of *which* decision was
made and why, leaving no decision trail. Kept as a complement, not a substitute.

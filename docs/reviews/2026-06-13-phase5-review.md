### gateway

- **[CRITICAL] mintedNonce stored but never verified — fabricated nonce bypasses single-use guarantee**
  - `/Users/iam/Work/Projects/aisy-harness/packages/core-ts/src/gateway/index.ts:303-341`
  - Why: The defect is confirmed by direct code inspection. issueCard mints a UUID as card.mintedNonce (line 295) but handleCardTap never compares tap.nonce against card.mintedNonce. The tapKey is "${tap.cardId}|${tap.nonce}", so consumedTaps deduplicates only the (cardId, caller-supplied-nonce) pair. Any caller who knows the cardId (visible in the Telegram callback) can pass an arbitrary fresh nonce string and will clear all checks: the staleness guard only rejects nonces containing "expired"/"stale" or past expiresAt; the actionHash check only requires echoing the stored hash (also visible in the callback); the replay check only blocks the exact same nonce string used before. Additionally, the card is never deleted from the cards Map after a successful confirmation (line 340 only adds to consumedTaps), so an attacker can confirm the same card an unlimited number of times by supplying a different fabricated nonce each time. The spec (§5.3 step 1) explicitly requires "nonce N exists AND state == issued", and ADR-0029 point 4 states "a tap only applies to the exact pending action it was issued for" via the minted nonce — that binding is entirely absent. The AC-02-7 and AC-02-8 tests do not catch this because they use a caller-fabricated nonce ('nonce-001') that issueCard never returned, and they happen to pass because the missing check is the one being bypassed.
  - Fix: In handleCardTap, immediately after fetching the card, add: `if (tap.nonce !== card.mintedNonce) throw new NonceStale(tap.cardId)`. This ensures the tap can only succeed when the caller echoes the exact nonce that was minted and embedded in the card payload. Additionally, after the successful confirmation at line 340, delete the card from the map: `cards.delete(tap.cardId)` — this ensures that once confirmed, no further nonce (minted or fabricated) can ever re-confirm the same card. The corresponding tests in gateway.spec.ts must also be updated: AC-02-7 must obtain the mintedNonce from the card (requiring either exposing it through issueCard's return value or through a test-seam), and the fabricated 'nonce-001' strings replaced with the real minted nonce.
- **[CRITICAL] Card never removed from Map after confirmation — different nonce bypasses replay guard**
  - `/Users/iam/Work/Projects/aisy-harness/packages/core-ts/src/gateway/index.ts:340`
  - Why: The attack scenario concretely survives scrutiny. After a successful tap, `consumedTaps.add("cardId|nonce-001")` is called but `cards.delete(tap.cardId)` is never called. - **[HIGH] streamReply checks outbound lockout only once; spec §5.2 requires re-checking on each lockout event mid-stream**
  - `/Users/iam/Work/Projects/aisy-harness/packages/core-ts/src/gateway/index.ts:267-277`

If Safety fires `outbound.lockout.set` mid-stream (e.g., because an untrusted span arrives concurrently while a reply is being streamed), the remaining tokens continue flowing unchecked — directly violating the lockout's purpose of preventing information exfiltration. The `GatewayDeps` interface exposes `isOutboundLocked(): boolean` as a poll function, so per-iteration polling is the available implementation strategy. The AC-02-6 test only exercises the pre-stream case (`isOutboundLocked: () => true` from construction), so it passes today and would continue to pass even after the fix — the mid-stream scenario has no test coverage at all. The defect is real: the spec requirement is unambiguous, the failure scenario is concrete, and the fix is trivially available using the existing deps contract.
  - Fix: Move the lockout poll inside the `for-await` loop so every token iteration re-checks the lockout state. Also re-check `isSafetyAvailable()` at the same point to maintain the fail-closed invariant for mid-stream Safety loss:

```ts
async streamReply(_chatId: number, tokens: AsyncIterable<string>): Promise<void> {
  if (deps.isOutboundLocked()) {
    throw new OutboundBlocked()
  }
  if (!deps.isSafetyAvailable()) {
    throw new TransportError('Safety unavailable — outbound fails closed')
  }
  for await (const _token of tokens) {
    if (deps.isOutboundLocked()) {
      throw new OutboundBlocked()
    }
    if (!deps.isSafetyAvailable()) {
      throw new TransportError('Safety unavailable — outbound fails closed')
    }
    void _token
  }
}
```
